# Помощь рядом — описание бизнес-процессов и правил

Документ описывает **что делает платформа**, **кто и при каких условиях** может выполнять действия, и **как устроены жизненные циклы** заявок, верификации и финансов. Техническая реализация (API, стек, инфраструктура) вынесена в код и OpenAPI.

---

## 1. Назначение платформы

**Помощь рядом** — платформа, которая связывает людей, нуждающихся в помощи (**бенефициары**), с **волонтёрами** и **партнёрами** (организациями/модераторами, проверяющими заявки).

Два вида помощи:

| Вид | Суть | Кто участвует |
|-----|------|----------------|
| **Материальная** | Сбор денег на конкретную цель; после трат — отчёт и закрытие | Бенефициар, волонтёры (жертвователи), партнёр |
| **Социальная** | Волонтёрская работа в назначенное время и месте; после — отчёт | Бенефициар, волонтёры (исполнители), партнёр |

Деньги по материальным заявкам проходят через платформу: сбор → хранение до вывода бенефициаром → отчёт о расходах → проверка партнёром.

---

## 2. Участники и зоны ответственности

### 2.1. Бенефициар (получатель помощи)

Человек или представитель группы (пожилые, малоимущие, многодетные семьи, инвалиды, одинокие родители и др.), которому нужна помощь.

**Может:**
- зарегистрироваться по телефону и пройти верификацию;
- указать **способы связи** для общения с волонтёрами (см. раздел 13);
- создавать и вести **свои** материальные и социальные заявки;
- редактировать заявку **до одобрения партнёром** и **после возврата на доработку**;
- отменять заявку на ранних этапах;
- прерывать материальную заявку в фазе сбора;
- запрашивать вывод собранных средств (материальная помощь);
- подтверждать актуальность социальной заявки, запускать и завершать работу;
- формировать отчёт по завершённой помощи;
- получать уведомления о статусах заявок, верификации, пожертвованиях.

**Не может:**
- видеть чужие заявки;
- модерировать;
- жертвовать деньги (это роль волонтёра).

### 2.2. Волонтёр

Пользователь, который помогает деньгами и/или личным участием.

**Может:**
- зарегистрироваться по телефону и пройти верификацию;
- указать **способы связи** для общения с бенефициарами (см. раздел 13);
- жертвовать на **материальные** заявки в статусе сбора;
- просматривать опубликованные **социальные** заявки и **присоединяться** к ним (с учётом навыков);
- отказываться от участия до старта работ;
- получать уведомления о социальных событиях (отмена заявки, старт/финиш работ).

**Не может:**
- создавать заявки на получение помощи;
- модерировать заявки и верификации.

### 2.3. Партнёр (модератор)

**Сотрудник платформы** (представитель доверенной организации). Проверяет людей и содержание заявок/отчётов. **Публичного профиля нет** — волонтёры и бенефициары не просматривают карточку партнёра и не связываются с ним через «контакты в профиле».

**Может:**
- входить в систему по **электронной почте** (аккаунт создаётся платформой, не через саморегистрацию);
- просматривать очереди: заявки на модерации, отчёты, верификации;
- одобрять / отклонять / вернуть на доработку заявки и отчёты;
- прерывать заявки и отменять (в рамках правил инициатора);
- одобрять, отклонять и **отзывать** верификации;
- для материальных отчётов — **сверять фактически потраченную сумму** с заявленной;
- при модерации заявки или верификации видеть **служебный контакт заявителя** (телефон аккаунта и способы связи из профиля бенефициара/волонтёра);
- получать уведомления о новых объектах в очередях модерации.

**Не может:**
- создавать заявки от имени бенефициара;
- загружать документы верификации за пользователя;
- публиковать способы связи для других пользователей приложения.

### 2.4. Администратор

**Сотрудник платформы.** Сопровождает справочники (каталог навыков волонтёров: код, название, нужна ли верификация навыка, нужны ли подтверждающие материалы). Получает те же уведомления о модерации, что и партнёр.

- Вход по **электронной почте**; аккаунт создаётся **в особом порядке** (как у партнёра), без саморегистрации.
- **Публичного профиля нет**; контакты для волонтёров и бенефициаров не предусмотрены.

### 2.5. Роль «Организация»

Зарезервирована на будущее, в текущих процессах не участвует.

### 2.6. Кто инициировал действие (для аудита)

- **Владелец заявки** — бенефициар;
- **Партнёр** — модератор;
- **Система** — автоматические переходы (например, недостаточно волонтёров, истечение сроков).

## 3. Регистрация, вход и базовый доступ

### 3.1. Саморегистрация (бенефициар и волонтёр)

- Регистрация только для ролей **бенефициар** и **волонтёр** по **номеру телефона** (один номер — один аккаунт).
- Базовые данные: ФИО, пол, дата рождения, город.
- Номер подтверждается **SMS-кодом**; этот телефон используется для **входа**, но **не показывается** другим бенефициарам и волонтёрам в публичном профиле (см. раздел 13).

### 3.2. Вход (бенефициар и волонтёр)

- Вход по **телефону** и паролю с повторным подтверждением SMS.
- Сессия с ограниченным временем жизни; возможен выход с одного устройства или со всех.

### 3.3. Партнёр и администратор (без саморегистрации)

- Аккаунты **партнёра** и **администратора** создаются **платформой** (особый порядок), пользователь сам не регистрируется в общем потоке.
- Вход по **электронной почте** и паролю (без SMS на телефон).
- Публичных профилей и «контактов для связи» в приложении у этих ролей **нет** — это внутренние учётные записи сотрудников.

### 3.4. Удаление аккаунта

- Пользователь может удалить свой аккаунт (мягкое удаление с сохранением аудита по политике хранения данных) — в рамках политики для своей роли.

### 3.5. Общее правило доступа к операциям

**Создание заявок и вывод средств** доступны только **верифицированному** бенефициару. До верификации пользователь может заполнить профиль и подать заявку на проверку личности. **Верифицированные** бенефициар и волонтёр должны иметь **не менее одного способа связи** для общения на платформе (раздел 13).

## 4. Верификация личности

### 4.1. Зачем нужна

Подтверждает, что пользователь — реальный человек, а данные о категории нуждаемости (для бенефициара) или навыках (для волонтёра) соответствуют правилам платформы.

### 4.2. Процесс для бенефициара

1. Общие данные (ФИО, пол, дата рождения, город) — можно уточнить при верификации.
2. Документы: паспортные данные, фото с документом.
3. **Категория** получателя помощи (одна из: инвалидность, малоимущие, многодетная семья, одинокий родитель, пожилые и др.) — для каждой категории свой набор полей и подтверждающих документов (справки МСЭ, о доходах, о составе семьи и т.д.).
4. **Способы связи** — не менее одного канала для общения с волонтёрами (например Telegram, email, WhatsApp; **не** телефон входа в аккаунт). Указываются при верификации, далее редактируются в профиле.
5. Превью и отправка **на проверку партнёру**.

**Правила:**
- Одновременно может быть только **одна активная попытка** на проверке.
- После **одобрения** сменить категорию самостоятельно нельзя — нужна новая верификация.
- Партнёр может **отклонить** (нужна новая попытка) или **отозвать** уже одобренную верификацию (статус пользователя сбрасывается; у волонтёра при отзыве снимаются активные записи на социальные заявки).

### 4.3. Процесс для волонтёра

- Упрощённая форма: общие данные, документы, **навыки** из каталога (часть навыков требует подтверждённой верификации).
- **Способы связи** — не менее одного канала, чтобы бенефициары могли связаться (телефон входа в публичный профиль **не попадает**).
- Та же цепочка: на модерации → одобрено / отклонено / отозвано.

Партнёр и администратор **не проходят** эту верификацию и **не заполняют** способы связи для публичного профиля.

### 4.4. Уведомления по верификации

| Событие | Кому |
|---------|------|
| Отправлена на проверку | Партнёры и администраторы |
| Одобрена | Бенефициар или волонтёр |
| Отклонена | Бенефициар или волонтёр |
| Отозвана после одобрения | Бенефициар или волонтёр |

## 5. Материальная помощь (заявка на сбор средств)

### 5.1. Содержание заявки

- Название и описание цели.
- **Категория** сбора (питание, лекарства, жильё и т.д.).
- **Запрашиваемая сумма** (целевой сбор).
- Фотографии (по желанию).

### 5.2. Жизненный цикл (статусы)

```
Ожидает модерации
    → Одобрена партнёром → Идёт сбор средств
    ↔ Возвращена на доработку (бенефициар правит и снова на модерацию)
    → Отклонена / Отменена (до сбора)

Идёт сбор средств
    → Собрана полная сумма
    → Прервана (бенефициар или партнёр — с причиной)
    → Отменена (до операционной фазы)

Собрана полная сумма
    → Ожидает отчёта (бенефициар тратит и отчитывается)
    → Просрочка отчёта (если не сдан в срок)

Ожидает проверки отчёта
    → Завершена (партнёр одобрил отчёт)
    ↔ Отчёт на доработке / отклонён
```

**Смысл статусов для пользователя:**

| Статус | Что видит бенефициар |
|--------|----------------------|
| Ожидает модерации | Заявка на проверке; можно отменить |
| Возвращена на доработку | Нужно исправить и отправить снова |
| Отклонена | Указана причина; можно править и отправить снова или отменить |
| Идёт сбор | Прогресс сбора, список жертвователей; можно прервать |
| Собрана полная сумма | Можно выводить средства и готовить отчёт |
| Прервана | Сбор остановлен; далее отчёт по факту |
| Ожидает отчёта | Нужно создать отчёт |
| Отчёт на проверке | Ждёт решения партнёра |
| Завершена | Успешное закрытие |
| Просрочка отчёта | Отчёт не сдан вовремя — ограничения по политике |

### 5.3. Ключевые бизнес-правила

1. **После одобрения партнёром** бенефициар **не может менять** текст, сумму и медиа заявки (только операции жизненного цикла: сбор, прерывание, отчёт).
2. **Редактирование** возможно в статусах «ожидает модерации», «возвращена на доработку», «отклонена» (после правок — повторная отправка на модерацию).
3. **Отмена** доступна бенефициару (и партнёру в рамках модерации) **до операционной фазы** сбора.
4. **Прерывание** — в фазе сбора; указывается причина; при наличии пожертвований заявка переходит в прерванное состояние с обязательностью дальнейшего отчёта по правилам MVP.
5. Заявка **автоматически** считается полностью профинансированной, когда сумма пожертвований достигает запрошенной.
6. **Пожертвования** принимаются только от **верифицированных** волонтёров на **материальные** заявки в подходящем статусе.

### 5.4. Действия партнёра по заявке

- Одобрить → открывается сбор.
- Отклонить → с указанием причины.
- Вернуть на доработку → бенефициар исправляет поля.
- Прервать / отменить → по регламенту инициатора.

### 5.5. Уведомления (материальная заявка)

| Событие | Кому |
|---------|------|
| Заявка отправлена на модерацию (создание или повтор после доработки) | Партнёры, администраторы |
| Одобрена / отклонена / на доработку / отменена / прервана | Бенефициар |
| Получено пожертвование | Бенефициар |

## 6. Социальная помощь (волонтёрское событие)

### 6.1. Содержание заявки

- Название, описание, **тип работы** (уборка, сопровождение, обучение и т.д.).
- **Обязательные и желательные навыки** (из каталога; часть подтягивается из типа помощи).
- Минимальное и максимальное число волонтёров.
- Дата, время начала, **длительность**.
- **Место проведения** (адрес/координаты на карте).
- **Взять с собой** — опциональный список (до 10 пунктов, до 100 символов на пункт): подсказки платформы по типу работы + свои формулировки бенефициара; фиксируется в версии заявки и виден волонтёрам после одобрения.
- Фото (по желанию).

Подсказки для поля «Взять с собой»: `GET /api/social-help-requests/items-to-bring/suggestions?category=...` (только бенефициар).

### 6.2. Жизненный цикл

```
Ожидает модерации
    → Набор волонтёров
    → Ожидание подтверждения актуальности (за N часов до старта)
    → Ожидание старта (волонтёры могут присоединяться)
    → В процессе (работа идёт)
    → Ожидает отчёта
    → Отчёт на модерации
    → Завершена

Отклонена / Отменена / Прервана — на соответствующих этапах
```

**Переходы, инициируемые системой** (фоновый планировщик Celery Beat, без HTTP-эндпоинтов для партнёра/админа):
- За **N часов до `start_at`** (precheck): при достаточном числе волонтёров → «ожидание подтверждения актуальности»; иначе → отмена.
- Истечение срока подтверждения актуальности → отмена.
- Недостаточно записавшихся волонтёров в «ожидании старта» → прерывание.
- Истечение таймаута старта без действия бенефициара → прерывание.
- Недостаточно пришедших к моменту старта (при старте бенефициаром) → прерывание.
- Волонтёры ушли во время работы → прерывание (по правилам бенефициара).

Параметры таймаутов: `SOCIAL_HELP_REQUEST_SCHEDULER__*` в `.env` (см. `.env.example`).

### 6.3. Участие волонтёров

| Статус участия | Смысл |
|----------------|--------|
| Записался | Волонтёр присоединился к заявке |
| Отказался до старта | Снял запись |
| Пришёл | Отмечен при старте работ |
| Не пришёл | Записан, но не явился |
| Ушёл после старта | Покинул работу до завершения |

**Правила:**
- Присоединение возможно, пока заявка в фазах набора / ожидания старта (до фиксации состава при старте).
- При **отмене** социальной заявки уведомляются волонтёры со статусом «записался».
- Навыки волонтёра должны покрывать **обязательные** навыки заявки (если навык требует верификации — волонтёр должен быть верифицирован с этим навыком).

### 6.4. Действия бенефициара

| Этап | Действие |
|------|----------|
| После одобрения | Ждёт набор волонтёров; может отменить |
| Перед событием | **Подтвердить** или **отклонить актуальность** (событие ещё нужно) |
| Время старта | **Начать выполнение** (фиксируется, кто пришёл) |
| После работы | **Завершить выполнение** → ожидание отчёта |
| Отчёт | Создать / обновить отчёт до одобрения |

**Редактирование заявки** — как у материальной: до одобрения и после возврата на доработку; **после публикации поля заявки не меняются**.

### 6.5. Уведомления (социальная заявка)

| Событие | Кому |
|---------|------|
| На модерации (создание / повтор) | Партнёры, администраторы |
| Одобрена / отклонена / доработка / отмена / прерывание | Бенефициар |
| Нужно подтвердить актуальность | Бенефициар |
| Волонтёр присоединился | Бенефициар |
| Заявка отменена | Записавшиеся волонтёры |
| Старт / завершение работ | Волонтёры, участвующие в событии |

## 7. Отчёты о выполненной помощи

### 7.1. Назначение

Подтверждают, что помощь оказана целевым образом: для материальной — **куда ушли деньги**; для социальной — **что сделано**.

### 7.2. Материальный отчёт

- Описание покупок / расходов.
- Подтверждающие фото (чеки, товары).
- Указывается **потраченная сумма**; партнёр при проверке **сверяет** её с собранным и правилами заявки (в т.ч. отдельный этап settlement review для очереди партнёра).

### 7.3. Социальный отчёт

- Описание выполненных работ.
- Подтверждающие материалы (фото).

### 7.4. Модерация отчёта (партнёр)

- **Одобрить** → заявка **завершена**.
- **Отклонить** → с причиной.
- **Вернуть на доработку** → бенефициар исправляет и отправляет снова.

Бенефициар может **редактировать отчёт**, пока он не одобрён (на модерации или на доработке).

### 7.5. Уведомления по отчётам

| Событие | Кому |
|---------|------|
| Отчёт отправлен на проверку | Партнёры, администраторы |
| Одобрён / отклонён / на доработку | Бенефициар |

### 7.6. Правило закрытия (MVP)

Без принятого партнёром **финального отчёта** заявка не переводится в окончательно **завершённое** состояние. Просрочка отчёта влечёт статус просрочки и последствия по операционной политике.

---

## 8. Финансы (материальная помощь)

### 8.1. Пожертвования

1. Волонтёр выбирает сумму и инициирует оплату.
2. Перенаправление на оплату у платёжного партнёра (в продакшене — ЮKassa).
3. При успешной оплате сумма **зачисляется** в заявку; бенефициар получает уведомление.
4. Повторная оплата с тем же ключом идемпотентности не дублирует пожертвование.

**Ограничения:** только материальные заявки в фазе сбора; волонтёр верифицирован.

### 8.2. Вывод средств бенефициаром

- Бенефициар **сам запрашивает** вывод (не автоматически при прерывании).
- Нужен **привязанный способ выплаты** (карта через процесс привязки).
- Сумма вывода **не больше** доступного остатка по заявке и **не больше** собранного.
- В MVP — **один итоговый отчёт** на заявку, а не отчёт на каждый вывод.

### 8.3. Прерывание и невыведенные средства (MVP)

- Бенефициар может **прервать** сбор и при необходимости **не выводить** остаток, если помощь больше не актуальна.
- **Невыведенные** после прерывания средства по правилам программы могут направляться в **резерв платформы**, а не бенефициару (фиксируется в оферте/политике).
- Если выведено больше, чем подтверждено отчётом, действуют правила **технического остатка** и возврата (пороги задаёт операционная политика).

### 8.4. Возвраты жертвователям

При сбое выплаты или по регламенту возможен **возврат** средств волонтёру.

### 8.5. Прозрачность для пользователя

По заявке отображаются: **запрошено**, **собрано**, **выведено**, **подтверждено отчётом**, **остаток**.

## 9. Уведомления

### 9.1. Назначение

Своевременно информировать участников о смене статусов и задачах, требующих действия (модерация, подтверждение актуальности, новое пожертвование).

### 9.2. Каналы

| Канал | Когда используется |
|-------|-------------------|
| **В приложении** | Список уведомлений, счётчик непрочитанных, отметка «прочитано» — всегда, для всех ролей |
| **Поток в браузере (SSE)** | Веб-интерфейс **партнёра и администратора**: новые уведомления приходят без опроса сервера; обновляются счётчик и списки очередей модерации |
| **Push на телефон** | Мобильные приложения бенефициара и волонтёра (и при необходимости модераторов), когда устройство зарегистрировано в приложении |

Бенефициар и волонтёр в мобильном клиенте полагаются на **push** и открытие экрана уведомлений; веб-модерация — на **SSE**, чтобы не делать частый polling.

## 10. Работа партнёра (очереди модерации)

Партнёр видит **сводку очередей** без ручного подсчёта:

- материальные и социальные заявки **на модерации**;
- отчёты **на модерации** (материальные и социальные);
- верификации бенефициаров и волонтёров **на проверке**;
- материальные отчёты, ожидающие **сверки расходов**.

Далее переходит в соответствующий список с фильтрами по типу и статусу.

- Удалить может **владелец** (пока файл не привязан к отправленной на проверку сущности — по правилам жизненного цикла сущности).

## 12. История изменений и аудит

### 12.1. Версии заявки

Каждое существенное изменение содержания или статуса заявки создаёт **новую версию**. Партнёр и бенефициар видят **историю версий** и **цепочку переходов статусов** (кто, когда, из какого статуса в какой).

### 12.2. Версии отчёта

Отчёт при доработке также версионируется; в истории сохраняются причины возврата и отклонения.

### 12.3. Верификация

Сохраняются попытки верификации и **снимок публичных данных** на момент одобрения (для последующей проверки без раскрытия всех чувствительных полей партнёрам вне процесса).

## 13. Справочники и профили

- **Города** — список для выбора при регистрации и в профиле.
- **Категории бенефициаров** — для верификации и материальных заявок.
- **Категории / типы помощи** — для заявок.
- **Каталог навыков волонтёров** — ведёт администратор; влияет на социальные заявки и профиль волонтёра.

Статистика для бенефициара и волонтёра: число заявок, участий, пожертвований и т.д. (в рамках своей роли).

### 13.1. Публичный профиль (только бенефициар и волонтёр)

У **партнёра** и **администратора** публичного профиля **нет**.

Другой пользователь (бенефициар или волонтёр) может открыть **публичный профиль** и увидеть, в частности:

- имя (эффективное отображаемое), город, аватар;
- для волонтёра — навыки из каталога;
- **способы связи** из раздела 13.2 (без документов верификации и без телефона входа в аккаунт).

Пример: волонтёр записался на социальную заявку → бенефициар открывает его профиль → видит Telegram или email → договаривается о времени и месте. Номер телефона, с которого волонтёр регистрировался, в профиле **не отображается**.

### 13.2. Способы связи (бенефициар и волонтёр)

| Что | Правило |
|-----|---------|
| Кто заполняет | Только **бенефициар** и **волонтёр** |
| Когда впервые | При **верификации** (шаг формы перед отправкой на проверку) |
| Сколько нужно | **Не менее одного** способа у верифицированного пользователя |
| Где менять дальше | **Профиль** (верификацию заново не открывают только ради смены контакта) |
| Типы | Email, Telegram, WhatsApp и др. (конкретный перечень — в продукте/приложении) |
| Телефон входа (SMS) | Для входа; **не** показывается в публичном профиле другим бенефициарам и волонтёрам |

Платформа **не подтверждает** доставку по email/Telegram/WhatsApp (нет кодов и ботов) — пользователь указывает каналы сам; при необходимости партнёр возвращает верификацию на доработку.

**Кто что видит:**

| Кто смотрит | Публичный профиль бенефициара / волонтёра | Служебный контакт при модерации |
|-------------|-------------------------------------------|----------------------------------|
| Бенефициар или волонтёр | Способы связи из п. выше, **без** телефона входа | — |
| Партнёр (заявка, верификация) | — | Телефон аккаунта заявителя **и** способы связи из профиля |
| Партнёр / админ как «публичный профиль» | **Нет** такого экрана | — |

Встроенного чата на платформе в MVP нет: координация — через указанные в профиле внешние каналы (мессенджеры, почта).

## 14. Границы и допущения MVP

- Один **финальный отчёт** на материальную заявку, не отчёт на каждый вывод.
- Нет **внутреннего кошелька** «баланс пользователя» в приложении — только привязка к заявке и вывод.
- Социальная помощь: автоматические отмены/прерывания по **недобору волонтёров** и **таймаутам** — без ручного вмешания партнёра.
- Роль **Организация** не реализована.
- Уведомления на телефон требуют настройки мобильного приложения (регистрация устройства после входа).
- **Чат** между пользователями не реализован; связь бенефициара и волонтёра — через способы связи в публичном профиле (раздел 13).

Техническое API: `/api/docs` (OpenAPI), название сервиса — **Помощь рядом API**.

Проверка состояния:

- `GET /api/health` — liveness (процесс отвечает).
- `GET /api/health/ready` — readiness: PostgreSQL (обязательно); Redis (при `REDIS__URL`); S3 — при заданных `bucket`, `access_key`, `secret_key` (endpoint опционален, для AWS может быть пустым); геокодер — реальный вызов активного провайдера (Yandex при API key, иначе mock); Firebase (если включён). Ответ **503**, если хотя бы одна обязательная интеграция недоступна. В prod Docker healthcheck использует `/ready`.

**Фоновые задачи (инфраструктура):** Redis + Celery worker + Celery Beat (`social.process_deadlines` — дедлайны соцзаявок). В dev дополнительно Flower (`:5555`, basic auth из `FLOWER_BASIC_AUTH_*`) для мониторинга очереди. Сервисы: `celery-worker`, `celery-beat`, `redis` в `docker/docker-compose.dev.yml` и prod compose.

При `NOTIFICATIONS__PUSH_ASYNC=true` FCM отправляется задачей `notifications.send_push` (запись в БД и SSE — синхронно в API).

При `AUTH__SMS_SEND_ASYNC=true` SMS-код подтверждения телефона (регистрация бенефициара/волонтёра, resend) отправляется задачей `auth.send_verification_sms` после commit в БД.

При `DONATIONS__WEBHOOK_PROCESS_ASYNC=true` webhook ЮKassa (`POST /api/donations/webhook/yookassa`) после проверки подписи ставит задачу `donations.process_payment_webhook` (до 5 retry). HTTP сразу отвечает **204**; при недоступности Redis — **503** (ЮKassa повторит доставку).

При `PAYOUTS__WEBHOOK_PROCESS_ASYNC=true` / `REFUNDS__WEBHOOK_PROCESS_ASYNC=true` — тот же паттерн для `POST /api/payouts/webhook/yookassa` (`payouts.process_payout_webhook`) и `POST /api/refunds/webhook/yookassa` (`refunds.process_refund_webhook`).

Ручной прогон без Beat: `poetry run manage run-social-deadlines`.
